Representational State Transfer (REST) is an increasingly popular software architecture that enables scalable, maintainable, and performant web services. An API exposes underlying data models the same way your browser makes a request to a server to receive a web page, through the HTTP protocol.

RESTful APIs allow client interaction via standard data objects and markup (i.e. XML, JSON, HTML); which is conceptually different than the representation of the data stored on the server. This means developers don't have to worry about dealing with conversions or advanced data manipulation to send and receive information, letting them focus on building a great front-end.

What makes an API "RESTful"?

For an entity to classify as RESTful it must satisfy the following constraints:

Client-Server Separation
This means the client is never concerned with data storage, similarly as the server is never concerned with the user-interface.
Stateless
The server never concerns itself with the state of the client; each request contains all the information needed to service the request.
Cacheable
A RESTful API will in some way state that it is cacheable or not, ensuring that data isn't stale or misused in your application
Layered System
A client cannot ordinarily tell whether it is connected directly to the end server, or to an intermediary along the way.
Code on Demand
Servers can temporarily extend or customize the functionality of a client by the transfer of executable code.
Uniform Interface
Ensures that the API will not fundamentally change it's architecture and developers can have confidence that an update won't break their applications.

Making Requests
Each request has a few key components that are required to be successfully serviced.

HTTP Method

Defining a request method tells the server what kind of action the client is attempting to make. With the right access, clients can Create, Retrieve, Update, or Delete server-side data (commonly referred to as CRUD operations) using the methods described below.

Name Opteration Description
GET Retrieve One of the most commonly used methods; allows you to do exactly as the name suggests. GET is what you will use to access any data objects that are already stored on the server. This method is read-only, meaning you will only be able to access information stored on the server, not manipulate it.
POST Create

Also a very commonly used Method in OrderCloud 3.0; allows you to send new objects to be created and stored on the server. With a POST Method call, you must also send an object as part of the request, called the "request body". The format of the request body will vary; however, in the context of OrderCloud the format is application/JSON in all cases but one.
PUT Create or Update

A utility method in RESTful API's. It allows you to create OR update an object stored on the server. With a PUT method call, you must also send an object in the request. If the ID of the object is not found on the server, a new object will be created with that ID. If the ID of the object is found, a PUT method will update the object with the new object you send in the request. Note that if you are using PUT to update a pre-existing object, you must send a full object in your request. This means you cannot send it an object with only the updates you wish to make.
PATCH Partial Update

Similar to a PUT in that it updates a pre-existing object already stored on the server. The difference is that with a PATCH method call, you only have to send the changes in the request object vs. the entire object. This works great for when you are only updating a small piece of the object you are targeting.
DELETE Delete

This will remove an entire object from server. You will not have to send an object with a DELETE method call, as the information of the target object you are attempting to delete will be in the parameters.

Request URL

The Request URL defines which API the client wishes to communicate with. The base URL is the part of the Request URL that identifies the location of the server the client is attempting to contact (this remains constant throughout a single API). The Routing URL informs the server what data the client is targeting, so the server knows where to perform the defined method.

HTTP Headers

Headers are synonymous with 'Meta Information', meaning they provide additional, descriptive (but necessary) information about the request. One typical use case of HTTP headers is sending authorization or bearer "tokens" for APIs with security measures. Another common header is the "Content-Type" which defines the format of the each request's body (if present) and response.

Parameters

Parameters are not necessarily a part of every request, because not every request requires a parameter. Parameters can be included in a request in two ways: in the routing URL or as query strings. Route parameters are denoted in most API documentation using a single bracket {} notation within the Routing URL. Query string parameters exist at the end of the Routing URL starting with /? each split up by &, using a key/value format of key=value.

Request Body

The final component is the request body. A request body, much like params, is required in some calls, while unnecessary in others. A Request Body is needed when when creating or updating an object on the server, while the exact opposite is true when retrieving or deleting an object. The format of the Request Body is defined within the "Content-Type" header and will match the response (when present).
